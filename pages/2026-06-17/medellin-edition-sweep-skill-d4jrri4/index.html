Executive Summary

Bottom line

The Medellin Edition Sweep is a recurring agent runbook for gathering useful local signals before a weekly Edition is drafted. It reads WhatsApp groups and deterministic scraper/source health, then produces candidate inputs for the newsletter: Move Smarter ideas, event picks worth considering, and source-onboarding proposals.

It is intentionally not infrastructure. There is no service, queue, or cron. A failed sweep must not block Thursday publishing because the nugget backlog buffers the weekly loop.

  • Use it before localcms edition draft <date> when you want fresher local intelligence.
  • Keep WhatsApp read-only and write only PII-stripped aggregate outputs.
  • Never edit the weekly Edition file during a sweep.

Where It Fits

The sweep is upstream of the weekly Edition. It creates evidence-backed inputs, but the Edition draft and operator approval still decide what ships.

1
Sweep
Read WhatsApp and source health; mine for patterns, event signals, and missing sources.
2
Backlog / flags
Append Move Smarter candidates, list selection-worthy event slugs, and propose source onboarding.
3
Draft
localcms edition draft <date> uses the event pool plus top backlog item.
4
Approve
Operator edits Signal, Shortlist, Move Smarter, and Personal Note.
5
Ship
Weekly email and IG carousel ship from the approved Edition.

What The Skill Actually Does

The runbook has three work lanes. Each lane has a constrained output so it remains useful without turning into a publishing system.

Lane 1

WhatsApp signal pass

Read-only scan of Medellin group activity.

Looks for
Recurring practical questions, repeated warnings, event links, venues, organizers.
Good signal
Recurrence across groups, not one-off anecdotes.
Constraint
Output must be PII-stripped and non-traceable to a member or message.
Lane 2

Scraper-source pass

Checks the deterministic source system and event candidate depth.

Looks for
Stale/erroring sources and whether the next Edition window has enough events.
Good signal
Healthy pool depth; live-regime pools are expected to be 100+ events/week.
Constraint
A thin pool means investigate upstream ingestion before Thursday.
Lane 3

Output synthesis

Turns raw signals into newsletter-ready inputs.

Writes
Append-only Move Smarter/Nugget backlog candidates.
Reports
Selection-worthy event slugs and source issues.
Does not do
No Edition edits and no automatic publication.

Outputs And Ownership

The sweep output is deliberately narrow. It should leave final editorial selection to the Edition loop.

Output Where it goes Owner after sweep Important constraint
Move Smarter / Nugget candidates apps/medellin-insider/content/editions/nugget-backlog.md Operator / weekly Edition editor Append to the end only; do not reorder or delete existing backlog items.
Selection-worthy events Final sweep report Edition drafting agent / operator List event slugs and why; do not edit Edition files during sweep.
Source proposals Handoff to localcms-scraper-site-onboarding Scraper/source onboarding workflow Use for events seen in chat but missing from the canonical store.
Failures / gaps Final sweep report Operator If WhatsApp or sources fail, report the failure and let Thursday continue from existing backlog.

Runbook Shape

  1. Confirm WhatsApp readiness

    Run the WhatsApp doctor and avoid concurrent sync. The local WhatsApp store supports one process at a time.

  2. Import or inspect a bounded read window

    Use a last-sweep date. The purpose is repeatable analysis, not indefinite scraping.

  3. Mine for recurrence and event signals

    Prioritize repeated practical patterns, missing event links, and recurring venues/organizers. One-off anecdotes should usually stay out.

  4. Check scraper sources and event pool depth

    Review source state and the next Thursday-to-Wednesday event window. Thin event pools are an ingestion problem, not a reason to lower the editorial bar.

  5. Write constrained outputs

    Append backlog candidates, list event slugs with rationale, hand missing-source candidates to source onboarding, and summarize failures.

Guardrails That Matter

Non-negotiables
  • Read-only WhatsApp: never send, react, or reply.
  • PII-stripped outputs only: no member names, message IDs, or traceable private quotes in backlog or Edition copy.
  • Append-only backlog behavior: add candidates at the end; the operator controls order.
  • Failed sweep does not block the weekly Edition.
  • Do not treat Facebook/WhatsApp/private chatter as automatically publishable evidence.

Practical Interpretation

What this means when actually using the skill.

Best use

  • Run it before drafting when you want fresher local intelligence.
  • Use it to keep Move Smarter grounded in repeated real questions.
  • Use it to spot events missing from the event database.

Weak use

  • Using it as a replacement for editorial judgment.
  • Treating one chat message as a newsletter claim.
  • Trying to publish directly from private-channel evidence.

Likely next improvement

  • Make the final sweep report format more explicit.
  • Clarify how last-sweep date is tracked.
  • Add a small example backlog candidate and event-flag example.

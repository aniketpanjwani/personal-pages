What Changed in This Revision

2026-06-15 access-ladder pass

This is a targeted revision of the 2026-06-14 doc, not a fresh brainstorm. The original through-line is preserved intact: one shared Navigator substrate, many host-tuned interfaces. Four new assumptions reshaped the rest:

(1) No-data-access is a launch/pilot/default privacy posture, not a permanent product boundary — future modes may inspect data, work with project files, write code, and run analysis. (2) The agent-brief mode (hand precise artifacts to the member's own Codex/Claude Code/R/Stata agent) is a valid first-class architecture, not a fallback. (3) Data-connected and code-writing modes are feasible higher-trust modes, not off-limits. (4) Codex Desktop's in-app Browser supports authentication, so an authenticated hosted Navigator inside Codex Browser is viable and worth optimizing for in the first pass. Together these add a sixth axis (access & trust posture), a new north-star idea — a progressive access/trust ladder (I2) — a vertical reframe of the recommended direction, a re-rank of the seven prior ideas, and the rescue of three previously-cut candidates as upper-rung modes. The revision is substantive and merited a rewritten doc (this file, in place).

Grounding Context

The Navigator already exists as one corpus rendered in two shells; the original design tension was whether those shells multiply or collapse into a shared substrate. This revision adds a second, orthogonal tension: how much access the Navigator is allowed — and treats today's no-data posture as the bottom rung of a ladder, not a wall.

The corpus. library/did-navigator/SKILL.md is an interview-driven decision tree encoding the Baker–Callaway–Cunningham–Goodman-Bacon–Sant'Anna DiD Practitioner's Guide. It interviews the economist one question at a time, classifies the design (2×2 / event study / staggered / extensions), walks an 8-step forward-engineering framework, and emits two artifacts: a design memo (decision + rationale + Guide citation per step) and an agent brief (a self-contained instruction block the economist hands to their own R/Python/Stata coding agent). The current non-goals — it does not run regressions, does not accept confidential microdata (it works from descriptions of data), and does not replace referee judgment — are now read as a default/launch posture: "your drafts, data, and code never leave your machine" is a deliberate promise to a scoop-paranoid profession, but the product is explicitly allowed to offer higher-trust modes later that inspect data, touch project files, write code, or run analysis, as opt-in steps up a ladder rather than a one-way door.

Shell A — the local skill (flagship). SKILL.md runs inside the economist's own Claude Code / Codex / Claude Desktop, on their machine, keys, and licenses — distributed under the Agent Skills open standard (Dec 2025; adopted by Claude Code, Codex, Copilot, Cursor, Gemini CLI, and 20+ hosts). This is the paid product's distribution model — and, critically, the shell that makes the upper rungs of the access ladder safe: data inspection and code-writing can happen local-first, on the member's machine, without anything leaving it.

Shell B — the hosted pilot. pilot/ is a deliberately tiny invite-only chat web app (assistant-ui front-end + pi-agent-core backend, one Bun process on a VPS) built to validate the corpus with real economists before charging. Hosted-as-product is explicitly rejected; full session traces are the "payment" that drives a confusion log and ≥1 material Navigator revision (the launch gate). Under this revision the hosted shell is the natural home for the low rungs (advisory, data-light) and the broad on-ramp.

The reachability picture (spike 2026-06-12, corrected 2026-06-15). The plan bet on the hosted pilot being driveable from the desktop apps' in-app browsers. The spike found Claude Desktop's in-app preview browser is origin-locked to its localhost dev server — a hosted https URL is silently unreachable in-app (cookies, typing, and streaming all work; navigation does not). Synthetic pointer clicks were also no-ops, so flows must complete without trusted pointer events. The Codex Desktop leg was skipped in the spike and was previously listed as unverified. Corrected assumption (take as fact): Codex Desktop's in-app Browser supports authentication — so an authenticated, session-carrying hosted Navigator inside Codex Browser is viable, and the first pass can optimize for it rather than for a public no-auth page. This removes the previous "Codex reaches https but carries no cookies/session" caveat and the tension it created with the invite-only pilot. The asymmetry that remains is one-sided: Codex = authenticated in-app hosted web works; Claude Desktop in-app preview stays localhost-locked, so the Claude side routes through the local skill or Claude-in-Chrome.

External signals. MCP Apps / SEP-1865 (Nov 2025) defines interactive ui:// resources inside MCP and mandates a text-only fallback for every UI-enabled tool — literally a "two shells, one tool" contract. Agent-native interface research (arXiv 2603.20300) frames the unit of design as an "invocable capability" with explicit schemas, not a human-facing feature — the memo and brief are capability contracts, and the access ladder is a contract about scope. The TurboTax pattern (live, accreting artifact) and SparkMe adaptive-interview research (rigid sequences are the #1 tedium source) bear on conversation behavior. LSP / compiler-linker are exact structural analogs for "one logic core, many thin clients," and capability-scoping / permission-tier models (least-privilege, progressive consent) are the analog for the access ladder. (assistant-ui is community-maintained with an uncertain trajectory; pi is niche — both are pilot-grade, not obviously scale-grade.)

  • 2 · Interview & conversation behavior — question mechanics, agent-drivability, adaptivity, guardrails.
  • 3 · Context-in — how drafts, data/repo descriptions, and prior work enter the session.

  • 5 · Continuity & return — sessions, state, resume, and corpus versioning over a paper's life.
  • 6 · Access & trust posture (new) — what the Navigator is allowed to see and do: descriptions only → agent-brief handoff → local file/data inspection → code-writing → running analysis. Data privacy as a tunable rung, not a fixed boundary.

Recommended Direction

The through-line

Two axes, one core. Keep the original insight as the horizontal axis — one substrate, many host-tuned renderers, not one monolithic UI and not N parallel reimplementations (Idea 1). Add a vertical axis this revision makes explicit: that same substrate climbs a progressive access/trust ladder (Idea 2). The logic is one adaptive core; the surfaces differ because the hosts differ; and the scope differs because trust is earned in rungs.

Concretely: the bottom rungs ship today — advisory (works from descriptions, data never leaves the machine) and the agent-brief handoff (the member's own coding agent inspects data and runs code; the Navigator never touches either). These are not a permanent ceiling — they are the launch/default privacy posture. Above them sit clearly-labeled higher-trust modes (the Navigator inspects local files/data, writes code, ultimately runs the analysis) that the architecture should not foreclose. The local-skill shell is what makes those upper rungs safe — local-first, on the member's keys, nothing leaving the box — while the authenticated hosted shell, now viable inside Codex Desktop's in-app Browser, is the low-friction entry rung for the data-light surfaces. The substrate (Idea 1) is what makes both the horizontal renderers and the vertical rungs cheap, and it is what makes the next Navigators (RDD, shift-share, SE/clustering) cheap too — so the architecture pays for itself across the whole Methods Navigator line. The schema-first brief (Idea 3) is the pivot rung where advisory becomes execution; Ideas 4–8 are what render, gate, and persist on top of that spine.

I1 Shared Navigator substrate + thin host-tuned renderers

Axis 1 · SubstrateConfidence 82%Complexity HighRecommended anchor
Description

Factor the Navigator into (a) a host-agnostic, serializable interview state machine plus a typed memo/brief schema, and (b) thin per-shell renderers. The agent emits typed "design-diagnostic" objects ({step, decision, rationale, guideCite, changeMyMind[], severity}); Claude Code renders them as AskUserQuestion + markdown, the CLI as numbered menus, the hosted shell as a live skeleton, and a future MCP host as a sandboxed ui:// resource — all from one emission. Delivered as a single MCP server tool, SEP-1865's mandated text-only fallback means the same tool lights up rich UI where the host allows and degrades to text where it doesn't. The same machine powers future Navigators (RDD, shift-share, SE/clustering) and the cross-method router that Step 8 ("name alternative designs rather than rescuing DiD") already implies — and, new in this revision, the serialized state is exactly what lets the same logic run at different access rungs (advisory ↔ data-inspecting ↔ code-writing) without a rewrite.

Basis

external: SEP-1865 mandates text-only fallback for all UI-enabled tools (modelcontextprotocol blog, Nov 2025); LSP and compiler/linker are exact structural analogs for one logic core + many clients (grounding). reasoned: N shells × M Navigators × K access rungs is a combinatorial integration trap if memo formatting and scope-handling live inside each prompt; a typed schema plus an explicit rung enum collapses it. direct: SKILL.md Step 8 already says to "name alternative designs," a dead-end sentence today but a live routing edge once a shared intake/memo schema exists.

Rationale

This is the highest-leverage structural decision in the whole product line and the literal answer to the framing question. It makes the "one corpus, many shells" promise architectural rather than aspirational, makes MCP a thin wrapper instead of a rewrite, makes Ideas 7 (live memo) and 8 (resume) nearly free because state is already a serializable document — and is the precondition for the access ladder (I2): you can only re-scope a core you can serialize.

Downsides

Highest up-front cost: today the product is a markdown SKILL.md, not code, so this is a real engineering investment before a paying member exists. SEP-1865 rich-UI host adoption (claude.ai, Desktop) is not confirmed shipping, so the ui:// payoff may lag the text-fallback payoff. Over-building the substrate before pilot feedback risks plumbing for a product that may still pivot — so scope the schema and the rung enum, not a framework.

I2 Progressive access / trust ladder (north star)

Axis 6 · Access postureNew this revisionConfidence 78%Complexity High (mostly directional)Recommended vertical axis
Description

Treat the Navigator's access posture as an explicit, opt-in ladder rather than a single fixed boundary, and use it as the product's north star for what to build next. The rungs: R0 Advisory — works from descriptions of data, nothing leaves the machine (today's default and the privacy promise). R1 Agent-brief handoff — the Navigator emits precise instructions/artifacts the member hands to their own Codex/Claude Code/R/Stata agent, which inspects data and runs code; the Navigator still never touches either (today, and the architecture in I3). R2 Local file/data inspection — running inside the member's own agent host, the Navigator reads project files, data schemas, or a sample locally, so it can stop asking facts it can see. R3 Code-writing — the Navigator authors the analysis code itself. R4 Run-for-you — the Navigator executes the analysis and reads results back. Each rung is opt-in, reversible, and labelled by precisely what crosses the machine boundary; the corpus and the typed memo/brief schema (I1) are constant across rungs — only scope changes. Present R2–R4 as a directional commitment that shapes today's decisions (e.g. don't bake in "the Navigator can never see a column name"), not as a near-term build.

Basis

direct: the user's stated direction — no-data-access is a "launch/pilot/default privacy posture, not a permanent product boundary," agent-brief mode is a valid architecture, and data-connected / code-writing are "feasible future higher-trust modes." SKILL.md already embodies R0–R1 (works from descriptions; emits an agent brief). external: least-privilege and progressive-consent permission tiers (security-capability model); agent-native research (arXiv 2603.20300) treats scope as part of the capability contract. reasoned: a scoop-paranoid economist will not jump from "shows you nothing" to "runs your data" in one step, but will climb a labelled ladder where each rung's exposure is legible and reversible — selling trust in rungs is how you eventually earn the high-value modes without spending the launch-day privacy promise.

Rationale

This reframes the single most strategically loaded question in the product — "does it touch data?" — from a yes/no wall into a roadmap. It protects the launch promise (R0/R1 stay the default and the marketing) while keeping the high-value modes (R2–R4) architecturally reachable, so a later pivot toward data-connected analysis is a rung you climb, not a rewrite. It pairs with I1 as the vertical axis to I1's horizontal one — together they define the product's whole design space — and it is what rescues three previously-cut ideas (economist-as-supervisor, reactive memo cells, PR-style ack loop) as upper-rung modes rather than dead ends.

Downsides

It is a direction, not a buildable unit — its near-term cost is restraint (don't foreclose the upper rungs) more than construction, and a north star with no shipped rung above R1 can drift into vaporware if it is allowed to justify scope creep before the pilot validates the corpus. Each real upper rung (R2+) reopens the privacy promise and needs its own consent UX, audit story, and per-host file-permission spec — so the ladder must stay clearly labelled "future" until the pilot's confusion log says the bottom rungs work. Higher rungs also collide with the "doesn't replace referee judgment" non-goal — running the analysis is closer to doing the econometrics than advising on it, which is a positioning decision, not just an engineering one.

I3 Schema-first agent brief + design.lock — the pivot rung

Axis 4 · Artifacts-outConfidence 86%Complexity MediumPromoted
Description

Emit the agent brief as a structured object first — target parameter, PT flavor, estimator, clustering level, aggregation, exact per-language package entry-points, expected data columns in the member's own variable names, outputs-to-check — with a human-readable rendering layered on top. The downstream coding agent parses the schema (idempotent, verifiable) instead of re-interpreting prose, so design decisions cannot be silently re-decided. Alongside it, emit a compact design.lock: every routing decision + the corpus git SHA + the per-choice Guide citation, so a coauthor, referee, or the member 18 months later can reproduce the identical design and verify which corpus version produced it. Carry the "what would change my mind" items forward as typed verification probes the brief states as acceptance criteria (e.g. "if the pre-trend slope exceeds X, reconsider PT-GT-all"). Re-read under this revision: the schema-first brief is the ladder's pivot rung (R1) — the exact contract that lets the member's own agent execute without the Navigator touching data — and the same typed object is what an R3 code-writing Navigator would consume internally. Building the brief as a schema now is what makes climbing the ladder cheap later.

Basis

direct: SKILL.md Phase 4 already enumerates the brief's exact fields, and Step 6's sensitivity hooks + the closing "what would change my mind" list are already conditional triggers; the pilot already stamps the corpus git SHA into every trace. external: agent-native interface research (arXiv 2603.20300) — the memo and brief are capability contracts, "make them schema-first"; lockfiles / Nix derivations / architect's stamps as the reproducibility-plus-attribution analog. reasoned: prose is lossy across the agent-to-agent boundary, where a "helpful" coding agent may re-decide the estimator and silently destroy the Navigator's value — and that boundary is precisely the R1 rung the ladder makes load-bearing.

Rationale

The handoff is the product's most critical seam and the rung most members will actually live on; a typed contract is the difference between "paste this and hope" and "your agent parses this and cannot drop a decision." The lock answers "why $20/mo, not $20 once" structurally — a scoop-paranoid economist won't share data but will share a version-pinned design, and a corpus revision (the pilot's whole purpose) becomes diffable against past designs. Promoted above the host-reach and memo-rendering ideas this revision because, under the access-ladder lens, it is the concrete anchor of the ladder's mid-rung rather than just one artifact among several.

Downsides

A maintained JSON schema for the brief introduces schema-drift risk as the Guide's recommendations evolve (versioning it with the corpus SHA mitigates this). The fully reactive version — results flowing back to re-evaluate the memo automatically — stays out of scope at R1 (it needs the R3/R4 rungs' execution capability), so the forward-only probes are the feasible half today. The lock is only as valuable as coauthors'/referees' willingness to open it.

I4 Smart intake: the on-ramp that climbs the ladder

Axis 3 · Context-inConfidence 85%Complexity Medium
Description

Let the economist seed the session from what they already wrote: paste a methods section / pre-analysis plan, or point the Navigator at a standardized repo file (did-intake.md) their coding agent fills from the actual data, in their own variable names. The Navigator pre-fills the intake checklist, marks inferred facts "I inferred X — correct me," and opens the interview only on genuine gaps and conflicts (a version-control "merge-base" over the draft). It compresses adaptively: a precise opening answer ("staggered, not-yet-treated comparison, Callaway–Sant'Anna with group-size weights") collapses to two or three confirmations, while a novice answer gets scaffolded. And it surfaces implementation-aware constraints at language-declaration time (e.g. the Python ecosystem's gaps for certain estimators) so the design is shaped around what the coding agent can actually run. Re-read under this revision: intake is the ladder's on-ramp — the did-intake.md-via-your-own-agent pattern already lives at R1, and the same intake step is what an R2 Navigator would satisfy by reading the file/data directly instead of asking. The interface should make "I'll describe it" vs "let my agent fill the file" vs (future) "let the Navigator look" feel like three rungs of one on-ramp, not three different products.

Basis

direct: SKILL.md Phase 1 already says to collect facts "skipping anything the member already volunteered," already "works from descriptions of data," and Phase 4 already anticipates Python coverage caveats ("say so in the brief rather than letting the downstream agent improvise"). external: SparkMe (arXiv 2602.21136, Feb 2026) finds rigid predetermined sequences are the #1 tedium source. reasoned: applied economists arrive from a half-finished methods section or a grant proposal, not a blank sheet — and the cheapest way up the access ladder is to let intake graduate from "describe" to "point at the file."

Rationale

Re-typing facts you have already written is the top reason an expert abandons an interview tool. Draft-in plausibly cuts intake time 60–80% for the common case and shifts the session's register from interrogation to refinement — which is what the $20/mo, expert-grade positioning requires. Surfacing ecosystem constraints early prevents the economist from committing to a design their agent can't implement and only discovering it at brief-reading time. Framing intake as the ladder on-ramp also gives the higher rungs a concrete, low-stakes first step (file-pointing) before any data is exposed.

Downsides

"Blind redaction" of identifying terms in a pasted draft is technically fuzzy (regex vs LLM pass; what counts as identifying?) — scope it down or drop it for an MVP and lean on the local-first shell where nothing leaves the machine anyway. Emergent adaptive compression is harder to test than a mode toggle and can over-compress past a load-bearing fact — it needs Idea 5's gates as a backstop. The implementation-constraint table needs a maintenance process as the R/Python/Stata ecosystems move.

I5 Pre-flight gates for load-bearing facts

Axis 2 · Interview behaviorConfidence 88%Complexity Low
Description

Recast the three named load-bearing facts — pre-treatment periods for the earliest-treated cohort, whether treatment timing varies, and whether anyone is never-treated — as hard gates the agent is not allowed to pass, rather than soft conventions. In a rich shell they render as a small gate panel; in text they're a numbered "BLOCKED ON" line. The agent refuses to proceed into the eight steps and says exactly which gate is open, the way a pre-flight or surgical checklist halts before the irreversible step.

Basis

direct: SKILL.md already states the doctrine verbatim — "Never assume a fact you can ask for… most commonly the number of pre-treatment periods, whether treatment timing varies, or whether anyone is never treated — ask and wait. Do not proceed on a guess." external: aviation pre-flight and surgical pre-incision checklists (Gawande) — the field made the small set of catastrophic-if-skipped items blocking and named.

Rationale

This makes an existing rule verifiable in traces rather than a behavior the model may silently drift past — and a credibility-destroying design built on a guessed pre-period is the single highest-cost failure the pilot's confusion log exists to catch. The agent's refusal-to-guess becomes a trust feature a careful economist will pay for. Lowest complexity of the set; pairs with Idea 4's compression as its safety floor — and matters more as the ladder climbs, since at R3/R4 an unblocked wrong fact gets baked into code that actually runs.

Downsides

Risk of over-rigidity — blocking on a fact the economist volunteered in a different phrasing; the agent must recognize equivalent information so the gate reads as competent, not pedantic.

I6 Authenticated Codex hosted web vs Claude skill/extension

Axis 1 · SubstrateConfidence 72%Complexity MediumReinterpreted
Description

Stop pretending the in-app browsers behave uniformly — but, given the corrected fact that Codex Desktop's in-app Browser supports authentication, optimize the first pass for an authenticated, session-carrying hosted Navigator inside Codex Browser rather than a public no-auth page. That single change dissolves the prior tension between "public reach" and the pilot's invite-only, passcode-gated model: the Codex surface can sit behind the same auth the pilot already uses (cookies/session work), keep the Enter-key-only, no-trusted-pointer constraint the spike requires, and serve the low rungs of the access ladder (R0 advisory, R1 brief) directly in-app. Route the Claude side through the local skill or Claude-in-Chrome / agent-browser, since Claude Desktop's in-app preview is localhost-locked. The interface adapts to each host's reachability envelope; the asymmetry is now Codex-favorable, not symmetric.

Basis

direct: the spike confirmed Claude Desktop's preview browser is origin-locked and that synthetic clicks were no-ops (pilot/spike/RESULTS-2026-06-12.md); the spike's Codex leg was skipped, and the corrected assumption (take as fact) is that Codex's in-app Browser supports auth. reasoned: the plan bet on uniform in-app driveability and the spike falsified it for Claude; honest agent-native reach means two first-class, host-tuned paths — an authenticated hosted surface for Codex, the local skill/extension for Claude — rather than degrading everyone to "just use a normal browser."

Rationale

It turns the spike's blocking finding into a deliberate two-surface design with a concrete, near-term distribution story for both major hosts — and the auth correction makes the Codex surface materially stronger than the prior revision assumed (session-carrying rather than cookie-less), which is what lets the hosted pilot double as the authenticated low-rung entry point of the access ladder.

Downsides

The Codex auth behavior is taken as fact here but was not exercised by the spike — a 10-minute Codex smoke test (re-runnable with the same mock) should still gate the build, since cookie/redirect handling inside the in-app Browser can have surprises. Demoted below the ladder, the brief, intake, and gates this revision: with the auth correction it is less of a load-bearing problem and more of an execution detail that sits under Idea 1's substrate and Idea 2's ladder, not a competing direction.

I7 Live accreting design memo (memo-as-canvas)

Axis 1 · SubstrateConfidence 82%Complexity Medium
Description

Render the 8-step memo as the primary, always-visible artifact from message one — a live skeleton with a classification banner and step-slots that fill or revise in place as the economist answers, TurboTax-refund-counter style — and demote the conversation to a thin advisory margin. The economist watches the design coalesce, can click a slot to reopen a decision, and catches a wrong classification before sinking eight steps on the wrong branch. The "what would change my mind" list becomes a persistent visible object, not a closing afterthought.

Basis

direct: SKILL.md's memo "sections mirroring Steps 1–8 plus the classification"; the pilot's transport already supports per-path set/append-text operations (pilot/web/src/main.tsx), and today's UI is a flat messages.map list with no memo surface — a real, transport-supported gap. external: the TurboTax live-artifact pattern (grounding).

Rationale

A live artifact converts an opaque interview into visibly-progressing co-authorship — the exact mechanism that makes structured-interview tools feel good rather than bureaucratic — and it should reduce the "I spent 30 turns and got a bad memo" abandonment the confusion log exists to catch. Because both shells render the same memo schema (Idea 1), the canvas is also a shared-substrate payoff; and with the Codex auth correction (Idea 6) it now has a viable in-app rich-host home, not just the local-dev preview.

Downsides

Real front-end restructuring of the pilot UI (flat list → canvas + margin), and deeper assistant-ui integration raises exposure to its uncertain maintenance trajectory. In a pure text shell (CLI / text-fallback MCP) "canvas" degrades to a re-printed memo block — still useful, but the visceral live-fill effect is a rich-host benefit. Sits below the ladder and its concrete rungs this revision: a rendering improvement, not a scope decision.

I8 Repo-local session state & resume (git-as-research-log)

Axis 5 · ContinuityConfidence 80%Complexity Medium
Description

Make a session a file, not server state: the skill writes .did-navigator/<paper-slug>/session.json (decisions so far, current step, open questions, corpus SHA) plus a rendered session.md into the economist's own repo after each exchange. Resuming means the skill reads the file and picks up mid-step — "Resuming your Medicaid study: you confirmed staggered adoption, county-year panel; I still need to know whether you observe any never-treated counties." No accounts, no passcode, no server session. The git history is the session log: diffable, branchable, and reviewable across a revise-and-resubmit cycle.

Basis

direct: the local-first promise ("your drafts, data, and code never leave your machine"); the pilot's localStorage session id (pilot/web/src/main.tsx) is the wrong, non-portable model for the product. external: the Agent Skills open standard has no native versioning or session-resume (grounding) — this is genuine whitespace. reasoned: economists work a paper over years and revisit the design memo during R&R 18 months later; a stateless skill forces a full re-interview.

Rationale

It makes the strongest version of the local-first promise real — even the session never leaves the machine — using a continuity model (git) this audience already trusts, and turns the Navigator from a one-shot consultant into a longitudinal collaborator. Newly relevant under this revision: writing files into the member's repo is the same capability primitive as the access ladder's R2 (local file/data inspection) — ship repo-local session state and you have already built the file-permission plumbing the upper rungs need, so this is the natural first step toward R2 that exposes nothing of the member's data.

Downsides

The skill shell doesn't persist files today; adding session writes means instructing each host to write to a path, and hosts differ in file-write permissions and conventions — it needs a per-host spec (the same spec the ladder's upper rungs will need). Stale-state resume must narrate prior decisions accurately or it erodes the trust it's meant to build.

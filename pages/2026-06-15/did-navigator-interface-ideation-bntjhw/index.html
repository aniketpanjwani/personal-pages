Grounding Context

The Navigator already exists as one corpus rendered in two shells, and the central design tension is whether those shells should multiply or collapse into a shared substrate.

The corpus. library/did-navigator/SKILL.md is an interview-driven decision tree encoding the Baker–Callaway–Cunningham–Goodman-Bacon–Sant'Anna DiD Practitioner's Guide. It interviews the economist one question at a time, classifies the design (2×2 / event study / staggered / extensions), walks an 8-step forward-engineering framework, and emits two artifacts: a design memo (decision + rationale + Guide citation per step) and an agent brief (a self-contained instruction block the economist hands to their own R/Python/Stata coding agent). Explicit non-goals: it does not run regressions, does not accept confidential microdata (it works from descriptions of data), and does not replace referee judgment. "Your drafts, data, and code never leave your machine" is a deliberate promise to a scoop-paranoid profession.

Shell A — the local skill (flagship). SKILL.md runs inside the economist's own Claude Code / Codex / Claude Desktop, on their machine, keys, and licenses — distributed under the Agent Skills open standard (Dec 2025; adopted by Claude Code, Codex, Copilot, Cursor, Gemini CLI, and 20+ hosts). This is the paid product's distribution model.

Shell B — the hosted pilot. pilot/ is a deliberately tiny invite-only chat web app (assistant-ui front-end + pi-agent-core backend, one Bun process on a VPS) built to validate the corpus with real economists before charging. Hosted-as-product is explicitly rejected; full session traces are the "payment" that drives a confusion log and ≥1 material Navigator revision (the launch gate).

The pivotal constraint (spike, 2026-06-12). The plan bet on the hosted pilot being driveable from the desktop apps' in-app browsers. The spike found Claude Desktop's in-app preview browser is origin-locked to its localhost dev server — a hosted https URL is silently unreachable in-app (cookies, typing, and streaming all work; navigation does not). Synthetic pointer clicks were also no-ops, so flows must complete without trusted pointer events. The Codex Desktop leg was skipped and remains untested. External docs (June 2026) show the opposite for Codex: its in-app browser can reach public https URLs (but carries no cookies/session). This asymmetry is load-bearing for any interface decision.

External signals. MCP Apps / SEP-1865 (Nov 2025) defines interactive ui:// resources inside MCP and mandates a text-only fallback for every UI-enabled tool — literally a "two shells, one tool" contract. Agent-native interface research (arXiv 2603.20300) frames the unit of design as an "invocable capability" with explicit schemas, not a human-facing feature — the memo and brief are capability contracts. The TurboTax pattern (live, accreting artifact) and SparkMe adaptive-interview research (rigid sequences are the #1 tedium source) bear on conversation behavior. And LSP / compiler-linker are exact structural analogs for "one logic core, many thin clients." (assistant-ui is community-maintained with an uncertain trajectory; pi is niche — both are pilot-grade, not obviously scale-grade.)

Topic Axes

  • 1 · Interface surface & substrate — which shells exist; one adaptive interface vs many; a shared logic core.
  • 2 · Interview & conversation behavior — question mechanics, agent-drivability, adaptivity, guardrails.
  • 3 · Context-in — how drafts, data/repo descriptions, and prior work enter the session.
  • 4 · Artifacts-out & handoff — the memo and agent brief, and how they land in the economist's coding agent / paper.
  • 5 · Continuity & return — sessions, state, resume, and corpus versioning over a paper's life.

Recommended Direction

The through-line

One substrate, many host-tuned renderers — not one monolithic UI, and not N parallel reimplementations. Treat the interview state machine plus the memo/brief schema (Idea 1) as the single source of truth, and make every surface a thin renderer over it. That directly answers the framing question "separate UIs or one adaptive interface?": the logic is one adaptive core; the surfaces are deliberately different because the hosts are genuinely different (Idea 2). Near term, render it three ways without a from-scratch rewrite — the local skill as the flagship (carrying the high-value behaviors: pre-flight gates, smart intake, schema-first brief, repo-local resume), a single MCP tool with the mandated text fallback as the broad-reach surface, and the hosted chat kept as the live-memo validation surface (Idea 3). The substrate is also what makes the next Navigators (RDD, shift-share, SE/clustering) cheap, so the architecture pays for itself across the whole Methods Navigator line. Ideas 4–7 are what render and behave on top of that spine.

I1 Shared Navigator substrate + thin host-tuned renderers

Axis 1 · SubstrateConfidence 80%Complexity HighRecommended anchor
Description

Factor the Navigator into (a) a host-agnostic, serializable interview state machine plus a typed memo/brief schema, and (b) thin per-shell renderers. The agent emits typed "design-diagnostic" objects ({step, decision, rationale, guideCite, changeMyMind[], severity}); Claude Code renders them as AskUserQuestion + markdown, the CLI as numbered menus, the hosted shell as a live skeleton, and a future MCP host as a sandboxed ui:// resource — all from one emission. Delivered as a single MCP server tool, SEP-1865's mandated text-only fallback means the same tool lights up rich UI where the host allows and degrades to text where it doesn't. The same machine powers future Navigators (RDD, shift-share, SE/clustering) and the cross-method router that Step 8 ("name alternative designs rather than rescuing DiD") already implies.

Basis

external: SEP-1865 mandates text-only fallback for all UI-enabled tools (modelcontextprotocol blog, Nov 2025); LSP and compiler/linker are exact structural analogs for one logic core + many clients (grounding). reasoned: N shells × M Navigators is a combinatorial integration trap if memo formatting lives inside each prompt; a typed schema collapses it to N + M. direct: SKILL.md Step 8 already says to "name alternative designs," which is a dead-end sentence today but a live routing edge once a shared intake/memo schema exists.

Rationale

This is the highest-leverage structural decision in the whole product line and the literal answer to the framing question. It makes the "one corpus, many shells" promise architectural rather than aspirational, makes MCP a thin wrapper instead of a rewrite, and makes Ideas 3 (live memo) and 7 (resume) nearly free because state is already a serializable document.

Downsides

Highest up-front cost: today the product is a markdown SKILL.md, not code, so this is a real engineering investment before a paying member exists. SEP-1865 rich-UI host adoption (claude.ai, Desktop) is not confirmed shipping, so the ui:// payoff may lag the text-fallback payoff. Over-building the substrate before pilot feedback risks plumbing for a product that may still pivot — so scope the schema, not a framework.

I2 Asymmetry-aware reach: Codex-public vs Claude-skill/extension

Axis 1 · SubstrateConfidence 65%Complexity Medium
Description

Stop pretending the in-app browsers behave uniformly. Ship a public, no-auth, URL-stateful, Enter-key-only hosted page for Codex Desktop's in-app browser (which reaches public https but carries no cookies/session), and route the Claude side through the local skill or Claude-in-Chrome / agent-browser, since Claude Desktop's in-app preview is localhost-locked. The interface adapts to each host's reachability envelope.

Basis

direct: the spike confirmed Claude Desktop's preview browser is origin-locked and that synthetic clicks were no-ops (pilot/spike/RESULTS-2026-06-12.md); grounding's external docs show Codex's in-app browser reaches public https with no cookies. reasoned: the plan bet on uniform in-app driveability and the spike falsified it; honest agent-native reach means two first-class, host-tuned paths rather than degrading everyone to "just use a normal browser."

Rationale

It dissolves the spike's blocking finding instead of working around it, and it gives the product a concrete, near-term distribution story for both major hosts — the renegotiation the spike flagged becomes a deliberate two-surface design.

Downsides

The Codex leg is still unverified — it rests on external docs, not the (skipped) spike; a 10-minute Codex smoke test should gate this. A public no-auth page is in direct tension with the pilot's invite-only, passcode-gated validation model — reconcile (e.g. token-in-URL, or accept this only post-pilot) rather than glossing it. This is a near-term reach tactic that should sit under Idea 1's substrate, not compete with it.

I3 Live accreting design memo (memo-as-canvas)

Axis 1 · SubstrateConfidence 82%Complexity Medium
Description

Render the 8-step memo as the primary, always-visible artifact from message one — a live skeleton with a classification banner and step-slots that fill or revise in place as the economist answers, TurboTax-refund-counter style — and demote the conversation to a thin advisory margin. The economist watches the design coalesce, can click a slot to reopen a decision, and catches a wrong classification before sinking eight steps on the wrong branch. The "what would change my mind" list becomes a persistent visible object, not a closing afterthought.

Basis

direct: SKILL.md's memo "sections mirroring Steps 1–8 plus the classification"; the pilot's transport already supports per-path set/append-text operations (pilot/web/src/main.tsx), and today's UI is a flat messages.map list with no memo surface — a real, transport-supported gap. external: the TurboTax live-artifact pattern (grounding).

Rationale

A live artifact converts an opaque interview into visibly-progressing co-authorship — the exact mechanism that makes structured-interview tools feel good rather than bureaucratic — and it should reduce the "I spent 30 turns and got a bad memo" abandonment the confusion log exists to catch. Because both shells render the same memo schema (Idea 1), the canvas is also a shared-substrate payoff.

Downsides

Real front-end restructuring of the pilot UI (flat list → canvas + margin), and deeper assistant-ui integration raises exposure to its uncertain maintenance trajectory. In a pure text shell (CLI / text-fallback MCP) "canvas" degrades to a re-printed memo block — still useful, but the visceral live-fill effect is a rich-host benefit.

I4 Smart intake: ingest the draft, ask only the gaps

Axis 3 · Context-inConfidence 85%Complexity Medium
Description

Let the economist seed the session from what they already wrote: paste a methods section / pre-analysis plan, or point the Navigator at a standardized repo file (did-intake.md) their coding agent fills from the actual data, in their own variable names. The Navigator pre-fills the intake checklist, marks inferred facts "I inferred X — correct me," and opens the interview only on genuine gaps and conflicts (a version-control "merge-base" over the draft). It compresses adaptively: a precise opening answer ("staggered, not-yet-treated comparison, Callaway–Sant'Anna with group-size weights") collapses to two or three confirmations, while a novice answer gets scaffolded. And it surfaces implementation-aware constraints at language-declaration time (e.g. the Python ecosystem's gaps for certain estimators) so the design is shaped around what the coding agent can actually run.

Basis

direct: SKILL.md Phase 1 already says to collect facts "skipping anything the member already volunteered," already "works from descriptions of data," and Phase 4 already anticipates Python coverage caveats ("say so in the brief rather than letting the downstream agent improvise"). external: SparkMe (arXiv 2602.21136, Feb 2026) finds rigid predetermined sequences are the #1 tedium source. reasoned: applied economists arrive from a half-finished methods section or a grant proposal, not a blank sheet.

Rationale

Re-typing facts you have already written is the top reason an expert abandons an interview tool. Draft-in plausibly cuts intake time 60–80% for the common case and shifts the session's register from interrogation to refinement — which is what the $20/mo, expert-grade positioning requires. Surfacing ecosystem constraints early prevents the economist from committing to a design their agent can't implement and only discovering it at brief-reading time.

Downsides

"Blind redaction" of identifying terms in a pasted draft is technically fuzzy (regex vs LLM pass; what counts as identifying?) — scope it down or drop it for an MVP and lean on the local-first shell where nothing leaves the machine anyway. Emergent adaptive compression is harder to test than a mode toggle and can over-compress past a load-bearing fact — it needs Idea 5's gates as a backstop. The implementation-constraint table needs a maintenance process as the R/Python/Stata ecosystems move.

I5 Pre-flight gates for load-bearing facts

Axis 2 · Interview behaviorConfidence 88%Complexity Low
Description

Recast the three named load-bearing facts — pre-treatment periods for the earliest-treated cohort, whether treatment timing varies, and whether anyone is never-treated — as hard gates the agent is not allowed to pass, rather than soft conventions. In a rich shell they render as a small gate panel; in text they're a numbered "BLOCKED ON" line. The agent refuses to proceed into the eight steps and says exactly which gate is open, the way a pre-flight or surgical checklist halts before the irreversible step.

Basis

direct: SKILL.md already states the doctrine verbatim — "Never assume a fact you can ask for… most commonly the number of pre-treatment periods, whether treatment timing varies, or whether anyone is never treated — ask and wait. Do not proceed on a guess." external: aviation pre-flight and surgical pre-incision checklists (Gawande) — the field made the small set of catastrophic-if-skipped items blocking and named.

Rationale

This makes an existing rule verifiable in traces rather than a behavior the model may silently drift past — and a credibility-destroying design built on a guessed pre-period is the single highest-cost failure the pilot's confusion log exists to catch. The agent's refusal-to-guess becomes a trust feature a careful economist will pay for. Lowest complexity of the set; pairs with Idea 4's compression as its safety floor.

Downsides

Risk of over-rigidity — blocking on a fact the economist volunteered in a different phrasing; the agent must recognize equivalent information so the gate reads as competent, not pedantic.

I6 Schema-first agent brief + design.lock provenance

Axis 4 · Artifacts-outConfidence 86%Complexity Medium
Description

Emit the agent brief as a structured object first — target parameter, PT flavor, estimator, clustering level, aggregation, exact per-language package entry-points, expected data columns in the member's own variable names, outputs-to-check — with a human-readable rendering layered on top. The downstream coding agent parses the schema (idempotent, verifiable) instead of re-interpreting prose, so design decisions cannot be silently re-decided. Alongside it, emit a compact design.lock: every routing decision + the corpus git SHA + the per-choice Guide citation, so a coauthor, referee, or the member 18 months later can reproduce the identical design and verify which corpus version produced it. Carry the "what would change my mind" items forward as typed verification probes the brief states as acceptance criteria (e.g. "if the pre-trend slope exceeds X, reconsider PT-GT-all").

Basis

direct: SKILL.md Phase 4 already enumerates the brief's exact fields and Step 6's sensitivity hooks + the closing "what would change my mind" list are already conditional triggers; the pilot already stamps the corpus git SHA into every trace. external: agent-native interface research (arXiv 2603.20300) — the memo and brief are capability contracts, "make them schema-first"; lockfiles / Nix derivations / architect's stamps as the reproducibility-plus-attribution analog. reasoned: prose is lossy across the agent-to-agent boundary, where a "helpful" coding agent may re-decide the estimator and silently destroy the Navigator's value.

Rationale

The handoff is the product's most critical seam; a typed contract is the difference between "paste this and hope" and "your agent parses this and cannot drop a decision." The lock answers "why $20/mo, not $20 once" structurally — a scoop-paranoid economist won't share data but will share a version-pinned design, and a corpus revision (the pilot's whole purpose) becomes diffable against past designs.

Downsides

A maintained JSON schema for the brief introduces schema-drift risk as the Guide's recommendations evolve (versioning it with the corpus SHA mitigates this). The forward-only probes are feasible; the fully reactive version — results flowing back to re-evaluate the memo automatically — is deliberately out of scope (it needs an agent-to-agent back-channel the product doesn't have). The lock is only as valuable as coauthors/referees' willingness to open it.

I7 Repo-local session state & resume (git-as-research-log)

Axis 5 · ContinuityConfidence 80%Complexity Medium
Description

Make a session a file, not server state: the skill writes .did-navigator/<paper-slug>/session.json (decisions so far, current step, open questions, corpus SHA) plus a rendered session.md into the economist's own repo after each exchange. Resuming means the skill reads the file and picks up mid-step — "Resuming your Medicaid study: you confirmed staggered adoption, county-year panel; I still need to know whether you observe any never-treated counties." No accounts, no passcode, no server session. The git history is the session log: diffable, branchable, and reviewable across a revise-and-resubmit cycle.

Basis

direct: the local-first promise ("your drafts, data, and code never leave your machine"); the pilot's localStorage session id (pilot/web/src/main.tsx) is the wrong, non-portable model for the product. external: the Agent Skills open standard has no native versioning or session-resume (grounding) — this is genuine whitespace. reasoned: economists work a paper over years and revisit the design memo during R&R 18 months later; a stateless skill forces a full re-interview.

Rationale

It makes the strongest version of the local-first promise real — even the session never leaves the machine — using a continuity model (git) this audience already trusts, and turns the Navigator from a one-shot consultant into a longitudinal collaborator. It also makes the substrate's serialized state (Idea 1) directly useful and is the precondition for the rejected-but-natural branching extension.

Downsides

The skill shell doesn't persist files today; adding session writes means instructing each host to write to a path, and hosts differ in file-write permissions and conventions — it needs a per-host spec. Stale-state resume must narrate prior decisions accurately or it erodes the trust it's meant to build.

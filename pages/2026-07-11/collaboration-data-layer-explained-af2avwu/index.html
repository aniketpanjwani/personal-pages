The Collaboration Data Layer

The idea in one minute

Your product already keeps a research-grade provenance graph — who wrote which span, which agent proposed which suggestion, who accepted it. But it throws away almost all of the behavior around that provenance: compile attempts survive as a single overwritten flag, rejected agent work is deleted with its branch refs, nothing records how long a human deliberated over a suggestion, and no keystroke, view, or timing data exists anywhere.

The data layer fixes this with one posture: capture everything by default, cut back later. Behavioral data not captured today is gone forever; the option value of having it — for product decisions first, for essays and research as a byproduct — is the whole point.

The architecture follows a single slogan: own the spine, rent the lenses. The permanent record is an append-only event store in your own Postgres. PostHog and Sentry are bolted on as disposable viewing instruments — dashboards, session replay, exception triage — that never hold anything you can't lose.

Three properties matter more than the boxes:

  • No new service. Capture hooks live inside the services that already exist; the only new HTTP surface is a browser ingest endpoint and a Tier B upload route.
  • Isolation is structural, not a convention. A dedicated Postgres schema means backup exclusion, the read-only export role, and the purge exemption are one-line grants and flags — not hand-maintained table lists that drift.
  • The lenses are disposable. Nothing goes to a vendor that isn't derivable from the spine, and manuscript text never goes at all.

The one governing invariant

Invariant

Every recovery tool in this repo — reseed, restore, purge-with-snapshot-rewrite, projection repair — assumes derived state can be rebuilt by rewriting it. Telemetry is the one store that holds more history than the substrate it derives from, so recovery tooling may only ever append to it (boundary-marker events). Never renumber, rebuild, or overwrite.

This sentence is the deepest thing in the plan, and it came out of review: the architecture borrowed its "derived record" discipline from the op-log/git relationship, but the op-log can always be rebuilt from git. Telemetry can't be rebuilt from anything — a rejected suggestion's content, a keystroke's timing, a purged paper's ledger entry exist nowhere else. Four separate review findings (unstable op-log sequences, tombstone-free restores, database-dropping backups, gap-erasing markers) were all instances of rewrite-based tools colliding with a stream that must never be rewritten.

Life of an event

A mutation (save, suggestion, accept…)

1 · Commit to gitsource of truth, as today
2 · Op-log entrycarries a stable entry UUID
3 · Respond to clientmutation is done — telemetry never blocks it
4 · Event insertseparate transaction, after commit, best-effort
▼   and if step 4 ever fails…   ▼
Reconcilerperiodically diffs op-log entry-UUID sets against events and backfills gaps — lag, never lead

Two subtleties make this trustworthy:

  • Identity comes from the op-log entry UUID, not the row sequence. The op-log gets rewritten (and renumbered) by at least four admin paths. The UUID inside each entry survives every rewrite, so a backfilled event and the original event compute the same event_id and deduplicate instead of duplicating.
  • Backfill restores existence, not fidelity. Op-log entries carry no payloads, so a reconciler-created event is flagged as reduced-fidelity, and event classes the reconciler can't see (reads, browser batches, compiles) get outage loss counters. The record is honest about its own gaps.

A keystroke

The editor doesn't log literal key events — it logs CodeMirror changesets with per-edit timing offsets: every keystroke's effect and rhythm ("it's like seeing thought") without building a keylogger. Batches are client-stamped with a UUID, flushed on interval and on tab-hide, filtered to locally-originated transactions so a five-person live room doesn't record every edit five times, and sent with fetch keepalive — because sendBeacon, the obvious choice, can't carry an Authorization header. Server receive-time is the authoritative clock; client timestamps are only trusted as offsets within one batch.

The Tier B gate deserves one pedagogical note: it deliberately refuses to make decisions based on transcript content. Free text is untrusted model output — so the gate reads only structured signals (session cwd, paper ids and file paths in tool-call arguments), and anything reaching outside the registered repo skips the whole session. One skipped session is cheap; one leaked draft from a non-consenting coauthor is not.

The trust ledger

The product's pitch is "your paper is yours," and this layer records everything its users do. The plan resolves that tension with explicit, enforceable commitments rather than fine print:

Never leaves the spine

  • Manuscript text, in any form — the PostHog mirror is a schema-generated metadata allowlist
  • Session replay is default-deny masked (maskAllText), with an automated test that a fixture manuscript string never appears in a replay payload — if that test can't stay green, replay is dropped, not the invariant
  • Tier B transcripts, categorically
  • Tokens and emails — URLs scrubbed in both SDKs, opaque principal ids only, IPs discarded

Survives everything (by decision)

  • Telemetry survives paper purge and account deletion — your recorded, twice-affirmed call
  • The disclosure says so plainly, including that purged papers remain reconstructable, and that a purged paper's text can persist in other consenting users' transcripts
  • First capture is gated on a recorded acknowledgment of that disclosure — at account creation, share-link redemption, and credential minting, with a rollout step covering everyone who already exists
The deliberate inversion

Purge used to be trace-free: snapshots scrub tombstones, so nothing durable proved a paper was ever purged. Now telemetry is the purge ledger — purge writes paper_purge_started / paper_purge_completed events, and restore refuses to resurrect any paper whose completed purge event exists. Deleting the paper became the one thing the event store is required to remember.

Eight traps the reviews caught

The plan went through three deepening specialists and five document reviewers. These are the findings that changed it — each one is a small systems lesson. Click to expand.

Op-log sequences aren't stable identity data integrity

The first draft keyed events on the op-log's auto-increment sequence. But four admin paths delete and re-insert the whole op-log, renumbering everything — after one raw git push or one restore, every stored sequence dangles and the reconciler would mass-duplicate entire paper histories.

Fix: key on the entry UUID stored inside each entry's payload — the one identity that survives rewrites. Sequence becomes an ordering hint.

Partition keys break ON CONFLICT dedup postgres

Postgres requires every unique index on a partitioned table to include the partition key. Partitioning on insert-time received_at means a retried insert gets a different partition-key value — so the dedup constraint could never fire, and "duplicate insert is a no-op" was unimplementable as written.

Fix: partition on a deterministic event_ts assigned exactly once and carried through every retry; unique index is (event_ts, event_id); keep received_at as a separate clock column.

sendBeacon can't authenticate web platform

The classic flush-on-tab-close API can't set an Authorization header. The escape hatches — token in URL, cookie auth — each violate a house security rule (and cookie auth would introduce CSRF to a proxy that currently has none).

Fix: fetch(…, {keepalive: true}) — same unload semantics, real headers, same 64KiB budget, which the final flush now explicitly respects.

The backup pipeline drops the whole database ops

Existing backups use pg_dump --clean --create — the dump itself carries DROP DATABASE. Excluding the telemetry schema from the dump does nothing when restore drops the database it lives in: the "restore telemetry first" ordering would destroy telemetry at step two.

Fix: schema-scoped dumps restored into the existing database, and the backup script asserts the dump contains no DROP DATABASE statement.

A torn purge poisons the ledger forever crash semantics

One synchronous paper_purged event written before a long, interruptible snapshot-rewrite sequence means a crash mid-purge leaves an append-only ledger asserting something false — and append-only means it can never be corrected.

Fix: bracket the purge with started/completed events and make it resumable. Started-without-completed means "resume me," never "purged" and never "never happened."

Recovery markers turn lag into permanent loss reconciliation

"The reconciler only looks after the most recent marker" prevents mass re-backfill after restores — but if an outage gap exists when a marker is written, everything in the gap becomes permanently unreachable even though the data to backfill from still exists.

Fix: any tool writing a marker must first run a catch-up pass over pre-marker state. Markers schedule steady-state work; they never bury debt.

File paths are how cross-paper text leaks consent

Tier B's gate checked paper ids in tool-call arguments — but agents read files by path, and a session in repo A can freely read ~/papers/repo-B/draft.tex, uploading a non-consenting collaborator's manuscript while every id-based check passes.

Fix: resolve every path in structured arguments; any path outside the registered repo root skips the whole session.

Masking inputs doesn't mask a manuscript privacy

Session-replay "input masking" protects form fields — but CodeMirror renders the manuscript as ordinary DOM text, and it also appears in the PDF preview, suggestion diffs, and comments. Input masking alone would ship the paper to a vendor while the plan claimed otherwise.

Fix: default-deny (maskAllText) with a chrome-only unmask allowlist, held in place by an automated test — and if the test can't be kept green, replay is dropped rather than the guarantee.

Sequencing note

The plan assumes the realtime-collaboration branch (uncommitted in your main checkout) merges first: it introduces the transactional outbox, task-event tables, and additive-migration rules this spine extends. Its task events are mirrored into telemetry rather than merely consumed — under the realtime plan they're purged with the paper, and this layer must not lose the delegated-agent record when that happens.

Risks and what's deferred

Live risks

  • Trust bet — default-on capture with honest disclosure sits in tension with "your paper is yours." Mitigations: onboarding-bound disclosure, conservative agent-query scoping, content never leaving the spine. Watch the first design-partner reactions.
  • Realtime coordination — the plan extends uncommitted work; if it merges differently, the envelope adapts; if it stalls, the spine proceeds on current main and reconciles later.
  • Volume — changeset capture grows fast; partitions + verified-export-then-drop are the pressure valve, coarsening timing granularity is the named first lever.
  • Masking fidelity — the replay-masking test is the load-bearing privacy control; replay is dropped before the invariant ever is.

Deliberately deferred

  • The user-facing "your paper's story" dashboard — the schema is ready for it from day one; the UI comes later.
  • Experimentation/randomization hooks, formal research apparatus, streaming analytics.
  • LLM-based post-hoc task-type classification over the Parquet exports — the design guarantees it stays possible (agent intent fields + full payloads), so the eventual "acceptance rates by task type" analysis needs no capture-day taxonomy.

Still open (FYI-level, from review)

  • Rate limits on the agent query surface (the write path has them; the read path should mirror the pattern).
  • Where the capture token lives on a participant's machine (keychain vs. permissioned file).
  • Which container runs the DuckDB export (Postgres's port stays unpublished).
  • Whether the restore-time tombstone check fails closed when telemetry is soft-disabled (it should).

The answer in one minute

Overleaf is valuable collaboration infrastructure with increasingly costly seams

Executive synthesis

The dominant academic impression is not that Overleaf is a bad product. Academics keep using it because a coauthor can open a link, install nothing, compile the same project, and start editing immediately.

The pain begins when a paper stops being a self-contained document and becomes one output of a larger research system. Code, data, figures, tables, review metadata, git history, and AI agents then have to cross boundaries that Overleaf was not designed to make seamless.

For economists, the opportunity is clear: git-canonical and agent-native underneath, with Overleaf-level coauthor simplicity and human review above it.

A useful mental model

The manuscript is part of a research system

Economists do not merely write text. They move continuously between analysis, generated artifacts, manuscript claims, and coauthor review. Overleaf is strongest at the final step and weakest at the seams between steps.

1
Analysis
Code, data, models, and reruns produce evidence.
2
Artifacts
Tables, figures, and scalar results are generated elsewhere.
3
Manuscript
LaTeX lives in Overleaf or in a separate synced folder.
4
Collaboration
Coauthors comment, revise, compile, and negotiate wording.
Why academics stay

Overleaf solves coordination, not just typesetting

The strongest positive evidence converges on three benefits. Any replacement has to preserve all three at once.

Adoption

Zero setup

  • A collaborator opens a link and starts working.
  • No TeX installation, package management, or editor setup.
  • Works on managed university devices and across multiple machines.
Execution

One compile environment

  • Everyone sees the same packages and compiler behavior.
  • The PDF is always available without sending build artifacts.
  • Less-technical collaborators never debug a local toolchain.
Social moat

Coauthor fluency

  • Live editing, comments, sharing, and review are familiar.
  • No team-wide agreement on git is required.
  • The weakest technical collaborator remains productive.
Where the workflow breaks

Six recurring pain clusters

These problems are not evenly distributed. They become most visible on serious, long-lived, multi-author, computational papers.

Source of truth

Stale synchronization

Dropbox, GitHub, local files, and Overleaf can disagree.

  • Human edits can be overwritten by an agent working from an older copy.
  • Workarounds depend on timing and careful user behavior.
Research pipeline

Paper and analysis are separated

Figures and tables are generated outside the manuscript project.

  • Users copy outputs, maintain sync scripts, or create submodules and symlinks.
  • The PDF can drift from the code that produced its results.
Version control

Git is only a translation layer

Overleaf history is translated into commits on demand.

  • No branches, tags, LFS, or nested submodules.
  • Authorship and rename semantics do not behave like a normal repository.
Review

Comments do not cross boundaries

Local and cloud review metadata do not compose safely.

  • Git pushes can lose or displace comments and tracked changes.
  • Review provenance becomes fragile precisely when agents need it most.
Economics

Serious use becomes premium

The free plan is a weak fit for complex team projects.

  • Current free compile timeout: 10 seconds; premium: 240 seconds.
  • History, integrations, track changes, and larger collaborator counts are paid features.
Resilience

The browser is a dependency

Service incidents can become deadline incidents.

  • Writing, compiling, and syncing can depend on one hosted service.
  • Users are advised to keep local backups, but local work is not the default model.
What economists said

The economist evidence is unusually concrete

One public thread around Peter Ganong's lost edits captured the central tensions: coordination, safety, and the limits of today's hybrid workarounds.

Voice Observed view Product signal
Peter Ganong Claude overwrote manual edits because Dropbox had not synced; switching to git meant more coauthor coordination. The source-of-truth problem is real, but social migration cost dominates.
Rebecca Diamond Had experienced the same stale-sync problem; recommended making both human and agent edits locally. The failure is not isolated, and the current fix is behavioral discipline.
Scott Kominers Recommended routing both author and agent through Overleaf's git integration. Technical users can route around the problem, but the hybrid remains.
Stefanie Stantcheva Expected cloning through git or GitHub to keep human and agent in the same files. Users may assume the integration is more complete than it is.
Joe Vavra Wanted insulation between agents, active analysis, and manuscript text. Agent changes should be proposed and attributable by default.
Peter Ganong, again Preferred speed and direct editing despite the safety objection. Insulation should be a default, not a cage.
The deeper diagnosis

The problem is not “How can Claude edit a .tex file?”

Reframe

Joseph Steinberg praised agents for turning half-hour LaTeX chores into seconds, but still liked Overleaf for manual writing and collaboration. The friction was fitting the manuscript into a larger research repository without size limits, slow sync, copied assets, or a second paper-only repo.

The real problem is: How can the manuscript become a safe, collaborative view over a reproducible research repository?

Who the product must serve

Four economist user segments coexist on the same paper

Segment What they optimize What loses them
Collaboration-first pragmatist A link, a browser, and no coauthor setup Any workflow that requires teammates to learn git
Reproducibility-first git user One history for code, artifacts, and paper Lock-in, manual copying, weak diffs, and browser-only work
Agent-first hybrid user Fast automated edits with full project context Stale sync, multiple sources of truth, and missing provenance
Safety-first senior author Human control over manuscript claims Silent edits, automatic propagation, and unreviewed agent changes
What the evidence supports

A replacement needs parity above and differentiation below

Non-negotiable parity

Keep the social workflow effortless

  • Open a URL; edit, comment, and compile immediately.
  • Use one shared compile environment.
  • Share and manage permissions without git vocabulary.
  • Keep comments and suggestions durably anchored.
  • Survive deadline behavior and mixed technical ability.
Differentiation

Make the repository canonical

  • Paper, code, figures, tables, and build scripts share context.
  • Agents work in isolated branches and propose attributable changes.
  • Direct mode exists without erasing provenance.
  • Local, web, and agent edits share one history.
  • Offline work and full export are ordinary capabilities.
The product risk

Do not assume asynchronous collaboration is enough

Critical assumption to test

The current plan defers real-time keystroke collaboration. Yet effortless co-editing is one of Overleaf's clearest perceived advantages.

Economists may rarely edit the exact same paragraph simultaneously, but they do value opening the same paper during a call and working naturally without merge rituals. Observe real coauthor sessions before treating save-and-merge as sufficient.

Recommended validation

The next research should move from opinions to observed behavior

  1. Recruit across all four segments

    Interview 10–15 economists, making sure technically strong git users do not dominate the sample.

  2. Observe two real coauthor sessions

    Watch how authors divide work, compile, comment, and resolve simultaneous edits during an actual call or deadline push.

  3. Test the weakest-coauthor migration

    Give a browser-only collaborator a link and measure time to read, edit, comment, compile, and recover a prior version.

  4. Prototype both safety modes

    Compare suggestion-by-default with direct agent edits while preserving attribution in both modes.

  5. Run the two-hour replacement test

    Ask whether remaining resistance is product preference or only switching and coauthor coordination cost.

Executive Summary

What changed

The new instagram ingest pilot command gives Medellin Insider a local, audit-friendly Instagram ingestion lane. It reads existing LocalCMS Instagram source inventory, fetches recent posts through ScrapeCreators, preserves raw payloads, normalizes captions/media/lineage, creates deterministic classification results, and writes event-candidate scaffolds for Codex/operator review.

The important architectural decision is that this is not a production write path. It does not dispatch the Cloudflare agent worker, does not call /api/agent/save-event, and does not add Convex tables. It proves the operator workflow between source inventory and a future approved event write.

  • The deterministic layer intentionally over-selects; the review layer decides what can become an event.
  • The pilot showed exactly why that matters: broad social accounts produced out-of-market, past, duplicate, and recap posts.
  • The strongest ready candidates in the sample came from teatropablotobon, where captions and selected images provided enough event detail.

End-to-End Flow

The current workflow is a local review pipeline. It gathers evidence and stops at operator-approved candidates rather than writing directly into the common events database.

1
Source Inventory
CLI calls the existing LocalCMS instagram source list API for a source-owner account and active inventory.
2
ScrapeCreators Fetch
For each selected username, the command calls /v2/instagram/user/posts or reads a fixture payload.
3
Raw Preservation
One raw provider JSON file is saved per username under the run directory.
4
Normalization
The parser extracts post refs, URLs, timestamps, captions, media kind, image refs, and lineage.
5
Prefilter
A deterministic score labels posts as clearly_event, ambiguous, or clearly_not_event.
6
Candidate Scaffolds
Event-like posts become incomplete candidate objects marked as operator-review required.
7
Codex Review
Codex/operator review promotes, completes, rejects, or asks for more evidence.

Operator Command Path

  1. Run the pilot command

    bun --cwd packages/cli src/index.ts --env prod instagram ingest pilot --email <source-owner-email> --limit-sources 3 --posts-per-source 12 --since-days 30 --output json

  2. Choose scope explicitly

    Use repeated --source <username> flags for representative accounts, or --limit-sources for a bounded inventory pass. The command defaults to active sources only.

  3. Use fixtures when needed

    --fixture-dir <dir> reads <username>.json payloads instead of spending ScrapeCreators credits. This is how tests prove the artifact contract without network calls.

  4. Keep output repo-root relative

    The command now detects the repo root before resolving --out, so the standard bun --cwd packages/cli invocation still writes to docs/proofs/... at the repository root.

  5. Review before database writes

    Generated candidates are not events. They are structured review artifacts that can feed a later approved write path.

Parser And Normalizer Details

The parser is deliberately tolerant of provider shape changes. It extracts from common ScrapeCreators/Instagram payload layouts rather than assuming one exact schema.

Concern Current behavior Notes
Raw post array discovery Checks arrays at top-level payload, posts, data.posts, data.items, items, response.posts, and response.items. This keeps the local command resilient across ScrapeCreators response variants.
Post unwrapping If a post has a node, the parser uses node; otherwise it uses the post object directly. Matches the current hosted worker prompt expectation while tolerating flatter payloads.
Identity and links Uses id or pk, shortcode or code, direct url or permalink, otherwise builds https://www.instagram.com/p/<shortcode>/. Every candidate keeps source URL plus source ID, directory ID, raw path, and raw post index.
Captions Reads caption_text, string caption, caption.text, or edge_media_to_caption.edges[].node.text. Derived text is sanitized and previews truncate by Unicode code point so generated JSON remains parseable.
Media kind Infers reel, video, carousel, or image from product type, is_video, media_type, sidecar edges, and carousel media. Reels/videos are still caption-classified; image review is limited to available image refs.
Image refs Collects display URLs, thumbnails, image-version candidates, sidecar child images, and carousel media images. The local pilot references image URLs rather than downloading all images by default.
Date filtering Drops posts older than --since-days when timestamp evidence is available. Posts without timestamps are kept, because absence of provider timestamp should not silently erase evidence.

Classification And Candidate Rules

The deterministic layer is intentionally a prefilter. It is useful for narrowing review, but it is not allowed to create database events by itself.

Stage Rule Output
Evidence scoring Date evidence adds 2 points; time evidence adds 2; event terms add up to 3; image refs add 1; video/reel adds a reason but not a score. Score >= 5 becomes clearly_event; score >= 2 becomes ambiguous; otherwise clearly_not_event.
Date/time evidence Regexes detect relative days, weekdays, month names, Spanish dates, numeric dates, and common time formats. Evidence strings are kept in candidate JSON, but they are not final parsed event dates.
Candidate creation Every non-clearly_not_event post becomes a candidate scaffold. Candidate rows are marked needs_codex_review and include requiredBeforeDatabaseWrite.
Title/category guesses The first non-empty caption line becomes the title guess; category is inferred from terms like music, food, sports, party, workshop, and art. These are hints for review, not authoritative metadata.
Review gate Codex/operator review can mark rows ready, incomplete, needs-more-evidence, or rejected. Only reviewed and operator-approved rows should feed a future database write path.

How This Differs From The Cloudflare Agent Worker

Architecture boundary

The hosted Cloudflare worker path fetches Instagram posts inside the worker, builds a compact post summary, classifies candidates, optionally downloads images, and then saves events through /api/agent/save-event.

The local path deliberately stops earlier. It preserves raw evidence and candidate JSON in the repo, exposes lineage for review, and requires operator approval before any common-event write path. That makes it better for research calibration and safer while the Instagram event extraction rubric is still evolving.

  • Hosted worker: extraction agent plus callback write path.
  • Local pilot: evidence collection plus review artifacts only.
  • Current branch: preserves deterministic webpage scraping, edition sweep, and hosted worker code unchanged.

Known Limits And Next Improvements

Limit Why it matters Likely improvement
Broad accounts cross-post outside Medellin The pilot caught Buenos Aires and Cali posts from a source in the Medellin inventory. Add stronger city/venue heuristics and a city-confidence flag before candidate creation.
Past-event detection is basic Relative dates and old source posts can survive the first pass. Add a pilot-date-aware date parser before candidate scaffolding, while still keeping uncertain rows reviewable.
Images are referenced, not bulk-downloaded Some event flyers contain missing time/price details. Add an optional bounded --download-images or --image-review-limit mode for selected candidates.
No automatic database write path yet This is intentional for v1 safety, but approved candidates still need a write mechanism later. Add an explicit candidate import --approved-file ... command only after the review contract stabilizes.
No custom Codex subagents yet The current foundation is subagent-ready but not subagent-dependent. Once the rubric stabilizes, define narrow agents for triage, extraction, and lineage audit.
